# AI 에이전트용 Code Knowledge Index 증분 갱신 알고리즘 상세 설계서

본 문서는 **Code Knowledge Graph / Query Model**을 항상 최신 상태로 유지하기 위한 **증분 갱신(Incremental Update) 알고리즘**을 상세히 정의한다.

대상 독자는 **이 시스템을 구현하는 AI 에이전트**이며, 문서의 목적은 *구현 선택지를 열어두는 것*이 아니라 **결정론적 동작 규칙을 고정**하는 것이다.

---

## 1. 설계 목표

1. 전체 재분석(Full Rebuild)을 원칙적으로 금지한다.
2. 갱신 비용은 **변경된 파일 수와 그 영향 범위에 비례**해야 한다.
3. 갱신 중에도 Query API는 **일관된 스냅샷**을 제공해야 한다.
4. 파싱 실패 또는 오류 발생 시 **마지막 정상 상태를 유지**해야 한다.
5. 모든 갱신 결과는 **그래프 무결성 제약조건**을 만족해야 한다.

---

## 2. 핵심 개념 정의

### 2.1 변경 단위(Change Unit)
- 증분 갱신의 최소 단위는 **파일(File)** 이다.
- 메서드/클래스 단위 변경 감지는 허용되지 않는다.

### 2.2 영향 전파(Impact Propagation)
- 변경된 파일에서 정의된 심볼이
  - **호출하는 대상**
  - **호출당하는 대상**
  - **상속/구현 관계로 연결된 대상**

  에 대해 그래프 상에서 영향이 전파된다.

### 2.3 갱신 트랜잭션(Update Transaction)
- 하나의 변경 이벤트(Git push, commit)는 **단일 논리 트랜잭션**으로 처리된다.

---

## 3. 전체 증분 갱신 파이프라인

```
[Change Event]
      │
      ▼
[Changed File Resolver]
      │
      ▼
[Parse & Extract]
      │
      ▼
[Delta Graph Builder]
      │
      ▼
[Transactional Graph Update]
      │
      ▼
[Index Version Commit]
```

---

## 4. 단계별 알고리즘 상세

---

## 4.1 변경 파일 식별 단계

### 입력
- Git commit ID 또는 파일 시스템 이벤트

### 처리 규칙
1. 변경 유형을 분류한다.
   - ADD
   - MODIFY
   - DELETE

2. 변경 파일 목록을 생성한다.

### 출력
```json
{
  "commit_id": "string",
  "changed_files": [
    {
      "file_path": "string",
      "change_type": "ADD | MODIFY | DELETE"
    }
  ]
}
```

---

## 4.2 기존 그래프 영향 범위 산출

### 목적
- 변경 파일과 **그래프 상으로 연결된 기존 노드**를 선별한다.

### 알고리즘
1. 변경 파일이 정의하던 모든 심볼을 조회한다.
2. 각 심볼에 대해 다음 엣지를 따라 탐색한다.
   - calls (양방향)
   - inherits / implements
   - tests

3. 탐색 깊이는 기본값 1로 하며, 재계산 필요성이 있는 노드만 수집한다.

### 출력
```json
{
  "affected_node_ids": ["string"]
}
```

---

## 4.3 삭제 처리 규칙 (DELETE / MODIFY 공통)

### 원칙
- MODIFY는 **DELETE + ADD**의 합성 연산으로 처리한다.

### 처리 절차
1. 변경 파일이 정의한 모든 노드를 그래프에서 제거한다.
2. 해당 노드에서 출발하거나 도착하던 모든 엣지를 제거한다.

### 제약
- 다른 파일에서 정의된 노드는 삭제 대상이 아니다.

---

## 4.4 파싱 및 신규 노드 생성

### 처리 절차
1. 변경 파일을 파싱하여 AST를 생성한다.
2. AST를 공통 스키마로 변환한다.
3. 신규 노드를 임시 그래프(Delta Graph)에 생성한다.

### 실패 처리
- 파싱 실패 시:
  - 해당 파일의 갱신을 중단한다.
  - 기존 그래프 상태를 유지한다.
  - 오류는 로그로만 기록한다.

---

## 4.5 엣지 재구성 알고리즘

### 내부 엣지 (파일 내부)
- 파일 내부에서 정의된 심볼 간 관계는 **항상 전면 재계산**한다.

### 외부 엣지 (파일 간)
- 다음 규칙을 따른다.

1. 변경 파일 → 외부 심볼
   - 신규 calls / depends_on 엣지 생성

2. 외부 심볼 → 변경 파일
   - 기존 엣지 중 삭제된 심볼을 가리키던 엣지만 제거

### Reflection / Dynamic Call
- 해석 불가능한 호출은 `dynamic=true`로 표시한다.

---

## 4.6 메트릭 재계산 규칙

- 변경 파일에 정의된 노드만 재계산 대상이다.
- 영향을 받은 외부 노드는 fan-in / fan-out만 갱신한다.

---

## 4.7 트랜잭션 적용 단계

### 규칙
1. 모든 변경은 **단일 트랜잭션**으로 적용한다.
2. 중간 상태를 Query API에 노출하지 않는다.

### 순서
1. 삭제 연산 적용
2. 신규 노드 삽입
3. 엣지 삽입
4. 메트릭 갱신

---

## 4.8 버전 커밋 및 스냅샷 관리

### 규칙
- 모든 노드는 `last_updated_commit` 필드를 갱신한다.
- 이전 스냅샷은 롤백 가능 상태로 보존한다.

---

## 5. 동시성 및 경쟁 조건 처리

- 동일 파일에 대한 갱신은 직렬화한다.
- 서로 다른 파일 변경은 병렬 처리 가능하다.

---

## 6. 오류 및 예외 처리 정책

| 상황 | 동작 |
|---|---|
| 단일 파일 파싱 실패 | 해당 파일만 무시, 전체 갱신 계속 |
| 트랜잭션 실패 | 전체 롤백 |
| 그래프 무결성 위반 | 갱신 중단 및 롤백 |

---

## 7. 무결성 제약 조건 (Invariant)

1. 모든 GraphNode는 정확히 하나의 File에 의해 정의된다.
2. dangling edge는 존재할 수 없다.
3. 삭제된 심볼을 가리키는 엣지는 존재할 수 없다.
4. 동일 qualified_name을 가진 활성 노드는 하나만 존재한다.

---

## 8. 성능 상한 가이드

- 단일 파일 변경 시 갱신 시간은 500ms 이내를 목표로 한다.
- 변경 파일 수가 N일 때, 처리 시간은 O(N + 영향노드 수) 여야 한다.

---

## 9. 구현 시 강제 사항

- 증분 갱신 알고리즘은 전체 재색인을 호출해서는 안 된다.
- 변경 파일 외의 노드 메트릭을 전면 재계산해서는 안 된다.
- Query API는 항상 완결된 그래프 상태만을 반환해야 한다.

---

## 10. 결론

본 알고리즘은 **IDE 인덱서 수준의 안정성**을 목표로 하며, AI 에이전트가 신뢰 가능한 구조 질의를 수행할 수 있도록 한다.

증분 갱신은 성능 최적화가 아니라 **시스템의 정합성을 보장하는 핵심 기능**이다.
