# 1. 문서 목적
본 문서는 **AI 에이전트 전용 코드 지식 인덱싱 및 질의 도구**를 개발하기 위해 필요한 기능적·비기능적 요구사항을 정의한다. 본 도구의 주 사용자는 사람이 아닌 **AI 에이전트**이며, 인간 개발자는 해당 도구를 운영·관리하고 AI에게 제공하는 역할만 수행한다.

본 요구사항은 특히 **파일 변경 감지 및 자동 쿼리 모델 최신화 기능**을 핵심 확장 요소로 포함한다.

---

# 2. 시스템 개요

## 2.1 시스템 목적
- 대규모 코드베이스의 구조적 정보를 AI 에이전트가 즉시 질의·활용할 수 있도록 제공
- AI에게 프로젝트 구조를 반복 설명하지 않기 위한 **외부 구조 기억 장치(External Structural Memory)** 제공
- 코드 변경 시 자동으로 인덱스를 최신 상태로 유지하여 **항상 신뢰 가능한 질의 결과** 보장

## 2.2 사용자 정의
| 구분 | 설명 |
|---|---|
| AI 에이전트 | 본 도구의 유일한 실사용자. API 기반으로만 접근 |
| 인간 운영자 | 인프라 설정, 배포, 장애 대응 담당 |

# 4. 기능적 요구사항

## 4.1 코드 수집 및 변경 감지

### FR-1. 파일 변경 감지
- 시스템은 다음 중 하나 이상의 방식을 통해 코드 변경을 감지해야 한다.
  - Git webhook (push, merge, PR)
  - Git commit polling
  - 파일 시스템 이벤트 감시 (inotify 등)

### FR-2. 변경 범위 식별
- 변경 감지 시 다음 정보를 반드시 식별해야 한다.
  - 변경된 파일 목록
  - 변경 유형 (추가 / 수정 / 삭제)
  - 관련 커밋 ID 또는 타임스탬프

### FR-3. 증분 처리 원칙
- 전체 프로젝트 재분석을 수행해서는 안 된다.
- 변경된 파일 및 그로 인해 **의존 관계상 영향받는 심볼만** 재분석 대상에 포함해야 한다.

---

## 4.2 파싱 및 구조 분석

### FR-4. 언어 파싱
- 시스템은 tree-sitter 또는 동등 수준의 파서를 이용해 소스 코드를 AST로 변환해야 한다.
- 언어별 파서 구현은 어댑터 구조로 확장 가능해야 한다.

### FR-5. 공통 구조 스키마 변환
- 언어별 AST 결과는 반드시 **언어 독립적인 공통 스키마**로 정규화되어야 한다.
- 최소 포함 정보:
  - File / Class(Type) / Method(Function) / Field
  - Qualified name
  - 소스 위치 정보 (file, start line, end line)

---

## 4.3 관계 및 메트릭 분석

### FR-6. 관계 그래프 생성
- 다음 관계를 그래프 형태로 생성·유지해야 한다.
  - 포함 관계 (File → Class → Method)
  - 호출 관계 (Method → Method)
  - 상속 / 구현 관계
  - 테스트 연결 관계 (Test → Production Code)

### FR-7. 코드 메트릭 계산
- 최소한 다음 메트릭을 계산해야 한다.
  - LOC
  - Cyclomatic Complexity
  - Fan-in / Fan-out

---

## 4.4 쿼리 모델 저장 및 갱신

### FR-8. 그래프 기반 저장소
- 코드 구조 및 관계는 그래프 DB 또는 동등한 그래프 모델로 저장해야 한다.

### FR-9. 증분 갱신 규칙
- 변경된 파일과 연관된 노드 및 엣지만 삭제·갱신해야 한다.
- 갱신 과정은 트랜잭션 단위로 처리되어야 한다.

### FR-10. 버전 메타데이터
- 각 노드는 마지막 분석 시점의 커밋 ID 또는 버전 정보를 포함해야 한다.

---

## 4.5 AI 에이전트 전용 Query API

### FR-11. API 접근 방식
- 모든 기능은 **머신 친화적 API(JSON 기반)** 로만 제공해야 한다.
- 자연어 응답은 금지한다.

### FR-12. 필수 API 목록
- 심볼 조회 API
- 호출자 / 피호출자 조회 API
- 영향 범위 분석 API
- 복잡도 Hotspot 조회 API

### FR-13. 출력 스키마 고정성
- 동일 API는 항상 동일한 JSON 스키마를 반환해야 한다.
- 필드 누락 또는 포맷 변경은 허용되지 않는다.

---

# 5. 비기능적 요구사항

## 5.1 성능
- 변경 감지 후 쿼리 모델 갱신은 프로젝트 규모에 비례하지 않고 **변경 범위에 비례**해야 한다.
- 단일 API 응답 시간은 1초 이내를 목표로 한다.

## 5.2 신뢰성
- 파싱 실패 시 해당 파일만 격리 처리하고 전체 인덱스를 무효화해서는 안 된다.
- 마지막 정상 상태의 쿼리 모델을 유지해야 한다.

## 5.3 확장성
- 언어 파서, 메트릭, API는 플러그인 방식으로 확장 가능해야 한다.

## 5.4 보안
- 소스 코드 원문은 외부로 노출되지 않아야 한다.
- API 접근은 인증 토큰 기반으로 제한해야 한다.

---

# 6. AI 에이전트 사용 규약 (중요)

## 6.1 Tool Contract
- AI 에이전트는 본 시스템을 **읽기 전용 구조 지식 도구**로 사용한다.
- 에이전트는 원본 소스 전체를 직접 분석하지 않고 반드시 API를 우선 사용해야 한다.

## 6.2 권장 에이전트 워크플로우
1. 변경 요청 수신
2. 영향 범위 분석 API 호출
3. 관련 심볼 구조 파악
4. 수정 계획 수립
5. 코드 생성 또는 수정

---

# 7. 제외 사항 (명시적 비범위)
- 사람을 위한 시각적 UI 제공
- 자연어 기반 설명 생성 기능
- 실시간 IDE 플러그인 기능

---

# 8. 성공 기준
- AI 에이전트가 프로젝트 구조 설명 없이도 일관된 코드 수정 계획을 수립할 수 있을 것
- 코드 변경 후 쿼리 결과가 항상 최신 상태를 반영할 것
- 동일 질의에 대해 항상 결정론적 결과를 반환할 것

---

# 9. 요약
본 시스템은 IDE나 코드 뷰어가 아니라, **AI의 추론을 위한 코드 세계 모델**이다. 변경 감지와 증분 최신화는 선택 기능이 아닌 필수 조건이며, 모든 설계는 AI의 질의·추론 효율을 최우선으로 한다.
